# Клиент-серверное приложение для обмена сообщениями

Проект представляет собой клиент-серверное приложение на C++ с использованием библиотеки Boost.Asio для сетевого
взаимодействия. Сервер обрабатывает входящие запросы от клиентов, выполняет простую обработку (возвращает заранее
определённые ответы) и отправляет результат обратно. Клиент может отправлять два типа сообщений: `signal` (содержит
вектор комплексных чисел) и `information` (содержит количество ядер). Обмен данными осуществляется через кастомный
транспортный протокол на основе TCP.

### Краткое описание модулей

- **`clientserveriface`** – *абстрактные интерфейсы* `Network::Server` и `Network::Client`, а также фабрики (
  `ServerCreator`, `ClientCreator`) и структуры параметров (`ServerCreatorParams`, `ClientCreatorParams`). Определяет
  тип `ConnectionId` и стандартные обработчики событий:  
  `setNewConnectionHandler()`, `setCloseConnectionHandler()`, `setReadHandler()`.

- **`configuration_handler`** – загрузка/сохранение JSON-конфигов для сервера и клиента.  
  Методы: `getData(Configuration, User)` возвращает `json::object`; при отсутствии файла создаются значения по
  умолчанию.  
  *Пример конфига сервера:*
  ```json
  {"ports":[10000,10001], "multiConnect":[true,false], "magic":[123456,123456]}
  ```

- **`io_context_handler`** – обёртка над `boost::asio::io_context` с *автоматическим запуском рабочего потока*.
  Использует `executor_work_guard` для предотвращения остановки. Методы: `getIOContext()`, `start()`, `stop()`.

- **`logger`** – *потокобезопасный синглтон* для цветного вывода в `std::cout`. Уровни: `Trace`, `Debug`, `Info`,
  `Warn`, `Error`, `Critical`. Формат лога: время, ID потока, имя логгера, функция, сообщение.

- **`utils`** – общие алиасы (`error_code`, `tcp`, `json`, `CallbackError`, `PortInt`, `MagicInt`) и функции
  `ErrorHandler::check_error()` для единообразной обработки ошибок с логированием.

- **`connection`** – классы для TCP-соединений:
    - **`ConnectHandler`** – `connect()`, `disconnect()` (синхронно/асинхронно).
    - **`AcceptHandler`** – `accept()`, `close()`; настройка опций сокета (`reuse_address`).

- **`message`** – иерархия сообщений:
    - **`Message`** – базовый класс с полями `type` и `transaction`.
    - **`SignalMessage`** – содержит `central_Freq` и вектор `std::complex<float>`.
    - **`InformationMessage`** – содержит `numberCore`.
    - **`RawMessage`** – содержит вектор байт `data`.  
      Все реализуют виртуальный метод `serialize()` для преобразования в `json::object`.

- **`message_handler`** – преобразует `TransportMessage` (type + transaction + payload) в объекты `Message` и обратно.
  Использует фабрику `CreatorMessage`, где для каждого типа регистрируется функция создания.  
  Методы: `parse()`, `serialize()`.

- **`transport_handler`** – реализует транспортный протокол:  
  *магическое число (4 байта) + длина JSON (4 байта) + JSON-данные*.  
  Поддерживает синхронное/асинхронное чтение/запись. Извлекает из JSON поля `"type"` и `"transaction"`.  
  Колбэки: `setOnAllRead()`, `setOnAllWrite()`, `setOnError()`.

- **`session`** – объединяет `ConnectHandler`, `TransportHandler` и `MessageHandler` в единый интерфейс для работы с
  сокетом.  
  Методы: `connect()`, `accept()`, `read()`, `write()`, `disconnect()`.  
  Поддерживает синхронный и асинхронный режимы; позволяет установить `magicNumber`.

- **`session_manager`** – управляет множеством сессий на стороне сервера.  
  Использует `AcceptHandler` для приёма подключений, создаёт `Session` для каждого клиента и хранит их по
  `ConnectionId`.  
  Автоматически удаляет сессии при ошибках. Методы: `addSession()`, `getSession()`, `closeAcceptor()`.

- **`worker`** – пул задач с одним рабочим потоком.  
  Принимает `std::function<void()>` или `Task` (с полем `status`).  
  Управление: `addTask()`, `flush()` (дождаться выполнения всех задач), `stop(clearQueue)`.  
  Статусы задач: `Work`, `Success`, `Error`.

- **`request_response_handler`** – бизнес-логика обработки запросов:
    - `createRequest(type)` – создаёт запрос с `Transaction::Request`.
    - `createResponse(message)` – анализирует тип (`signal` / `information`), генерирует *случайные тестовые данные* и
      возвращает ответ с `Transaction::Response`. Для неизвестных типов возвращает сообщение об ошибке.

- **`segment_client`** – реализация `Network::Client`.  
  Подключается к серверу через `Session`.  
  Позволяет отправлять сообщения трёх типов: `signal`, `information`, `raw`.  
  Чтение выполняется в фоновом потоке через `Worker`. Полученные `raw`-ответы выводятся в лог.

- **`segment_server`** – реализация `Network::Server`.  
  Принимает подключения через `SessionManager`, для каждого клиента запускает отдельный `Worker` для чтения.  
  При получении запроса `signal`/`information` генерирует ответ через `request_response_handler`; `raw`-сообщения
  логируются.  
  Поддерживает несколько портов и опцию `multiConnect` (если `false`, то новый клиент отключает предыдущего).

---

> **Примечание:** модули построены по принципу *низкой связанности* и могут использоваться независимо. Все сетевые
> операции работают как в синхронном, так и в асинхронном режиме (управляется через `IOMode`). В текущей реализации
> клиента и сервера используется синхронный режим, но архитектура позволяет легко переключить его на асинхронный.

## Типы сообщений (JSON)

Все сообщения содержат поле `"type"`, а также `"transaction"` (0 – запрос, 1 – ответ, 2 – тесты, -1 – ошибка).

### Signal (сигнал)

**Запрос:**

```json
{
  "type": "signal",
  "transaction": 0
}
```

**Ответ:**

```json
{
  "type": "signal",
  "transaction": 1,
  "central_Freq": 6100,
  "signal": [
    [
      -88.659,
      -65.495
    ]
  ]
}
```

### Information (информация)

**Запрос:**

```json
{
  "type": "information",
  "transaction": 0
}
```

**Ответ:**

```json
{
  "type": "information",
  "transaction": 1,
  "numberCore": 4
}
```

### Raw (сырые данные)

**Запрос:**

```json
{
  "type": "raw",
  "transaction": 0,
  "data": []
}
```

**Ответ:**

```json
{
  "type": "raw",
  "transaction": 1,
  "data": [
    1,
    2,
    3
  ]
}
```

(массив байтов в десятичном виде)

## Сборка и запуск

### Требования

- **CMake** 3.21 или выше
- **C++17**-совместимый компилятор (GCC, Clang, MSVC)
- **Boost** (системная библиотека)
- **Google Test** (для тестов, опционально)

### Установка зависимостей

#### Ubuntu/Debian

```bash
sudo apt update
sudo apt install cmake build-essential libboost-all-dev
```

#### macOS (Homebrew)

```bash
brew install cmake boost
```

### Команды сборки

Проект собирается с помощью CMake из корневой директории. Используются следующие команды:

```bash
# 1. Конфигурация проекта (создание build-каталога)
cmake -B build -S . -DCMAKE_BUILD_TYPE=Release

# 2. Сборка проекта
cmake --build build --config Release -j$(nproc)

# 3. Установка (размещение исполняемых файлов и библиотек в папку ./Release)
cmake --install build --prefix ./Release --config Release
```

**Пояснения:**

- `-B build` – создаёт каталог `build` для выходных файлов сборки.
- `-S .` – указывает корень исходного кода (текущая директория).
- `-DCMAKE_BUILD_TYPE=Release` – устанавливает тип сборки `Release` (оптимизированный). Для отладки можно использовать
  `Debug`.
- `--config Release` – явно указывает конфигурацию для многоконфигурационных генераторов (например, Visual Studio). Для
  Make-генераторов этот параметр игнорируется.
- `-j$(nproc)` – параллельная сборка с использованием всех доступных ядер процессора.
- `--prefix ./Release` – каталог, в который будут установлены собранные файлы. После установки исполняемые файлы
  появятся в `./Release/bin/`, библиотеки – в `./Release/lib/`.

### Сборка с тестами

Если вы хотите собрать и запустить тесты, добавьте флаг `-DUNITTEST=ON` на этапе конфигурации:

```bash
cmake -B build -S . -DCMAKE_BUILD_TYPE=Release -DUNITTEST=ON
cmake --build build --config Release -j$(nproc)
```

если вы выполнили `cmake --install`). Запустить тесты можно командой:
После сборки тестовые исполняемые файлы будут находиться в каталоге сборки (например, `build/test/`) или в каталоге
установки (

```bash
cd build
ctest --output-on-failure
```

### Примечания

- Если вы используете **Windows** с генератором Visual Studio, убедитесь, что указаны правильные параметры `-A` (
  архитектура) и `-T` (инструментарий) при необходимости.
- При возникновении ошибок, связанных с отсутствием Boost, убедитесь, что переменная `BOOST_ROOT` установлена корректно
  или что Boost находится в стандартных путях системы.
- Для очистки сборки достаточно удалить каталог `build` и папку `Release` (если она была создана).
  Для включения тестов используйте флаг:

```bash
cmake -DUNITTEST=ON ..
make
```

### Конфигурационные файлы

#### Сервер (`server_config.json`)

```json
{
  "ports": [
    8080,
    8081
  ],
  "multiConnect": [
    true,
    false
  ],
  "magic": [
    123456,
    654321
  ]
}
```

- `ports` – массив портов, на которых сервер будет слушать (может быть несколько).
- `multiConnect` – разрешено ли несколько соединений на одном порту (соответствует каждому порту).
- `magic` – магическое число для каждого порта (используется в протоколе).

Сервер создаёт отдельный экземпляр `SegmentServer` для каждого порта.

#### Клиент (`client_config.json`)

```json
{
  "address": "127.0.0.1",
  "port": 8080,
  "magic": 123456
}
```

- `address` – IP-адрес сервера.
- `port` – порт сервера.
- `magic` – магическое число, которое должно совпадать с указанным на сервере.

## Открытие портов

Если сервер и клиент запускаются на **разных машинах**, необходимо открыть порт в брандмауэре и/или настроить проброс
портов на маршрутизаторе.

### Пример для Linux (UFW)

```bash
sudo ufw allow 8080/tcp
```

### Пример для Windows (в командной строке с правами администратора)

```cmd
netsh advfirewall firewall add rule name="MyServer" dir=in action=allow protocol=TCP localport=8080
```

**Внимание:** для доступа из интернета также может потребоваться проброс порта на роутере (NAT) и использование
публичного IP-адреса.

---

### Пример полного запуска

1. Создайте конфигурационные файлы (например, `server.json` и `client.json`).
2. Запустите сервер:
   ```bash
   ./server server.json
   ```
3. В другом терминале запустите клиент:
   ```bash
   ./client client.json
   ```
4. В клиенте введите сообщение, в сервере используйте команды для управления.

Убедитесь, что порты открыты (см. раздел «Открытие портов»).
Если аргументы не указаны, клиент попытается подключиться к `127.0.0.1:8000` (значения по умолчанию).

После запуска клиент предложит ввести тип сообщения:

- `signal` – отправить запрос сигнала.
- `information` – отправить запрос информации.
- `raw` – отправить сырые данные.

Сервер обработает запрос и вернёт соответствующий ответ, который будет отображён клиентом.

## Примечания по реализации

- Все сетевые операции могут работать как в синхронном, так и в асинхронном режиме (управляется через `IOMode`). В
  текущих реализациях сервера и клиента используется синхронный режим, но архитектура позволяет легко переключить.
- Магическое число используется для того, чтобы приёмник мог отличить валидные сообщения от случайных данных. Оно
  настраивается через конфигурацию.
- Обработка ошибок единообразна – через `ErrorHandler::check_error`, которая логирует ошибки и при необходимости
  возвращает `true` для критических кодов.
- Для управления задачами используется `Worker` – один поток на каждую логическую единицу (например, для сервера –
  отдельный воркер на приём, и по воркеру на каждого клиента для чтения).
- Сессии автоматически удаляются при ошибках чтения/записи или отключении клиента.
